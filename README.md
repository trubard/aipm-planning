# AIPM — 아이디어 검증 워크스페이스 (개인용)

> v2.4.2 · 2026-06-17
> **머릿속 아이디어 한 줄을, 검증된 기획서까지 끌고 가는 개인 작업 공간이다.**
> 떠오른 생각을 적으면 → 진짜 문제인지 검증하고 → 페르소나·PRD·피치덱까지 단계별로 채워준다.
> 손이 가는 일은 Claude Code **슬래시 명령어 7개**가 대신한다. 막힌 아이디어는 아카이브로 보내 교훈만 남긴다.

전부 마크다운(`.md`)이라 **GitHub 웹에서 바로 읽히고**, 폴더째 **Obsidian으로 열어도** 된다. 설치할 것도, 빌드할 것도 없다.

> 📖 **용어가 어렵게 느껴지면** → [용어집.md](structure/90_템플릿/용어집.md). 페인·워크어라운드·JTBD·HMW·North Star 등 이 시스템에 나오는 모든 전문용어를 *초보 기획자 눈높이*로, 한 가지 예시(편의점 신상 아이디어)로 통일해 풀었다. 막히면 여기부터.

## 60초 요약: 이렇게 돌아간다

아이디어는 **깔때기**를 따라 흐른다. 아래로 갈수록 좁아지고, 막히면 옆으로 빠진다.

```
   💡 한 줄 아이디어            🔍 진짜 문제인가?         📐 어떻게 만들까?        ✅ 출시 준비
  ┌─────────────┐          ┌──────────────┐        ┌──────────────┐
  │ 10_아이디어풀 │  ──검증──► │  20_탐색중     │ ──게이트─► │  30_검증정의   │ ──► 본개발/스핀아웃
  │ 백로그·문제풀  │          │ 페르소나·JTBD  │        │ PRD·MVP·피치덱 │
  └─────────────┘          └──────────────┘        └──────────────┘
                                  │
                                  └──► 🗄️ 99_아카이브  (접은 아이디어 + 死因·교훈)
```

| 단계 | 한 줄 설명 | 주로 쓰는 명령어 |
|---|---|---|
| 💡 **10 아이디어풀** | 생각을 한 줄로 쌓는 곳. 막연하면 *문제*부터 발굴 | `/문제발굴` |
| 🔍 **20 탐색중** | 진짜 문제·시장인지 검증하고 구체화 | `/아이디어검증` `/아이디어진행` |
| 📐 **30 검증정의** | 만들 것을 확정 — PRD·피치덱 | `/아이디어진행` |
| 🗄️ **99 아카이브** | 접은 아이디어 보관 (삭제 아님) | `/아이디어폐기` |

## 처음이라면, 이 순서대로

**0. 작업 공간 만들기** — 채팅창에 입력:
```
/초기화
```
> `my-idea/` 폴더가 생긴다. 여기에 내 모든 산출물이 쌓인다. (왜 두 폴더로 나뉘는지는 맨 아래 [폴더 구조](#폴더-구조-systemstructure--내-작업my-idea) 참고.)

**1. 아이디어를 한 줄로 적는다** — [my-idea/10_아이디어풀/아이디어_백로그.md](my-idea/10_아이디어풀/아이디어_백로그.md)에:
```
[누가—구체조건]가 [지금 하는 대안] 대신 [핵심 한 가지]로 [결과—숫자]
```
> 아직 막연해서 적을 게 없다면 → `/문제발굴 "<관심 영역>"`. 외주 사이트·앱 리뷰·커뮤니티·검색을 훑어 *사람들이 이미 돈·시간 쓰는 미해결 문제*를 찾아 [문제 인박스](my-idea/10_아이디어풀/문제_인박스.md)에 채워준다.

**2. 검증한다** — 이게 핵심:
```
/아이디어검증 "노인 대상 복약 알림 앱"
```
> 문제정의 → 경쟁 제품·유튜브 탐색 → 비즈니스 검증 → **Go / Pivot / Drop 판정**까지 한 번에. 결과는 `my-idea/20_탐색중/`에 저장된다.
> *(선택)* 가장 위험한 가정을 고객에게 직접 물어보려면 → `/고객인터뷰`(왜=정성) 또는 `/설문조사`(몇%=정량). 둘 다 The Mom Test 기반.

**3. 구체화하고 확정한다** — Go/Pivot이 나오면:
```
/아이디어진행 "복약알림" "HMW까지"     # 페르소나·JTBD·Journey·HMW
/아이디어진행 "복약알림" "PRD까지"     # 게이트 통과 후 NorthStar~PRD~피치덱
```
> 접을 거면 → `/아이디어폐기 "복약알림" "수요 신호 약함"`. 삭제하지 않고 교훈과 함께 아카이브로 옮긴다.

## 명령어 7개 — 치트시트

> 채팅창에 입력만 하면 된다. 인자가 비면 알아서 되묻는다.

| 명령어 | 한 줄로 | 결과물이 가는 곳 |
|---|---|---|
| `/초기화 [생성\|리셋]` | 작업 공간을 **만들거나** 싹 비워 **리셋**(리셋은 확인받음) | `my-idea/` |
| `/문제발굴 "<영역>"` | 미해결 문제를 웹서치로 찾아 **AI BM 6축**으로 태깅 | `10_아이디어풀/문제_인박스.md` |
| `/아이디어검증 "<한 줄>"` | 문제정의→경쟁탐색→비즈니스검증→**Go/Pivot/Drop** | `20_탐색중/<슬러그>/00_아이디어검증.md` |
| `/고객인터뷰 "<아이디어>"` | 가장 위험한 가정을 **인터뷰 대본**으로 (정성·왜) | 아이디어 폴더에 `인터뷰_가이드.md` |
| `/설문조사 "<아이디어>"` | 같은 가정을 **설문 문항**으로 (정량·몇%) | 아이디어 폴더에 `설문_문항.md` |
| `/아이디어진행 "<아이디어>" [목표]` | 현재 단계를 감지해 **다음 산출물**을 이어서 생성 | 단계 폴더에 번호순(`02_페르소나.md`…) |
| `/아이디어폐기 "<이름>" "<사유>"` | 접은 아이디어를 **사유·교훈·부활 트리거**와 보관(이동·보존) | `99_아카이브/<이름>/` |

### `/아이디어진행`의 목표(2번째 인자)

없으면 **다음 산출물 1개**만 만들고 멈춘다(검수 후 다시 호출). 어디까지 한 번에 갈지 지정할 수 있다:

```
/아이디어진행 "레시피봇"              # 다음 빠진 산출물 1개만
/아이디어진행 "레시피봇" "HMW까지"     # 20단계를 HMW까지 연속
/아이디어진행 "레시피봇" "PRD까지"     # 30단계로 승급해 1페이저 PRD까지
/아이디어진행 "레시피봇" "끝까지"      # 피치덱까지 (20→30 승급은 확인받음)
```

> **안전장치 3가지** — ① 세 명령어 모두 인자가 비면 먼저 되묻는다. ② `/아이디어폐기`는 폴더·산출물을 **절대 삭제하지 않는다**(이동·보존). ③ `/아이디어진행`은 **검증 없이 구체화하지 않는다** — `00_아이디어검증.md`가 없거나 Drop이면 멈춘다. **20→30 직행도 막는다** — HMW까지 끝나면 게이트 통과를 보고하고 확인받아 승급한다.

## 작성 규칙 (느슨하지만 지키면 편한 것들)

- **한 아이디어 = 한 폴더.** `20_탐색중/`·`30_검증정의/` 아래에 한글 아이디어명 폴더(공백 대신 붙여쓰기, 예: `장보기도우미`). 진행 상황은 [아이디어_백로그.md](my-idea/10_아이디어풀/아이디어_백로그.md) 표에서 한눈에.
- **한 줄 아이디어 양식**: `[누가—구체조건]`가 `[지금 하는 대안]` 대신 `[핵심 한 가지]`로 `[결과—가능하면 숫자]`. 원석은 양식 안 맞아도 일단 적는다.
- **이름·숫자는 고정.** 같은 아이디어 안에서 사용자 이름·핵심 숫자를 흔들지 않는다(예: 사용자=김OO 34세).
- **숫자로 끝낸다.** "편하게/많이/좋게"는 산출물이 아니다. 조건·기간·수치를 붙인다.
- **형식·순서**는 [기획_산출물_템플릿.md](structure/90_템플릿/기획_산출물_템플릿.md), 빈 양식은 [structure/90_템플릿/양식/](structure/90_템플릿/양식/). 명령어 없이 손으로 채워도 된다.

## 폴더 구조: 시스템(`structure/`) ↔ 내 작업(`my-idea/`)

저장소는 두 층으로 나뉜다. **시스템과 내 데이터를 섞지 않기 위해서다.** 보통은 `/초기화`가 알아서 처리하니 몰라도 된다 — 직접 손댈 때만 보면 된다.

- **`structure/`** — 공유·버전관리되는 **빈 골격(시스템)**. 단계 폴더 골격 + `90_템플릿/`(가이드·빈 양식). **읽기 전용**(양식은 여기서 읽기만). `git pull`로 업데이트되는 게 이 폴더다.
- **`my-idea/`** — 내 **실제 작업 공간**. `structure/`의 단계 폴더를 복제한 것으로, 모든 산출물이 여기 쌓인다. **`.gitignore`로 공유에서 제외**(개인 데이터는 로컬에만).

`/초기화` 대신 셸로 직접 해도 된다:

```bash
# 최초 1회 — structure에서 내 작업 공간 만들기
mkdir -p my-idea && cp -R structure/10_아이디어풀 structure/20_탐색중 structure/30_검증정의 structure/99_아카이브 my-idea/

# 전체 초기화 — 내 산출물 싹 지우고 빈 골격으로 (되돌릴 수 없음)
rm -rf my-idea && mkdir -p my-idea && cp -R structure/10_아이디어풀 structure/20_탐색중 structure/30_검증정의 structure/99_아카이브 my-idea/
```

> 7개 명령어는 모두 **산출물을 `my-idea/`에 쓰고, 양식은 `structure/90_템플릿/`에서 읽는다.**

## 업데이트 & 마이그레이션

각자 받아 쓰는 개인용이다. "배포"는 `git push`, "업데이트"는 `git pull`. 이력은 [CHANGELOG.md](CHANGELOG.md).

도구(명령어·양식) 변경은 대부분 **추가 위주**라 pull하면 새 기능이 그냥 생긴다. **내 작업물(`my-idea/`)은 `.gitignore`라 충돌이 없다.** 주의할 건 두 가지뿐:

| 상황 | 어떻게 |
|---|---|
| **새 기능 받기** | `git pull` — `.claude/commands/`·`structure/90_템플릿/`이 갱신된다. 명령어는 섹션 번호가 아니라 *내용*으로 찾으므로 옛 산출물도 그대로 호환. |
| **옛 산출물에 새 항목이 없음** | 양식이 바뀌어도 *이미 만든* 파일엔 새 항목이 자동 추가되지 않는다. 필요하면 그 아이디어에 명령어를 재실행하거나, 해당 섹션만 손으로 덧붙인다(강제 아님). |

> 전체를 빈 상태로 되돌리려면 위 "전체 초기화"로 `my-idea/`만 리셋한다(`structure/` 시스템은 유지).
